## (1) 문제 정의

### 문제 정의
일상에서 우리는 “조금 더 싸게”를 위해 이동·대기·준비 같은 시간을 추가로 쓰는 선택을 자주 합니다. 하지만 대부분의 의사결정은 **가시적인 지출(원)** 중심으로 이루어지고, **비가시적인 기회비용(시간)**은 과소평가되기 쉽습니다.

그 결과, 1,000원을 아끼기 위해 30분을 더 쓰는 선택처럼 **시간의 가치가 잠식**되거나, 결정 이후 “내가 잘 선택했나?”라는 막연한 후회가 남습니다. 또한 선택지가 많아질수록 비교 기준이 흔들리며 **의사결정 피로**가 누적됩니다. 더 나아가 이미 투자한 시간/노력 때문에 비합리적인 선택을 지속하는 **매몰 비용**(sunk cost) 상황도 발생할 수 있습니다.

이 프로젝트는 사용자의 시간 가치(시급)를 기준으로 선택지 A/B의 **총비용(직접비용 + 시간비용)**을 계산·비교하고, 결과와 함께 **계산 근거(중간값/수식)**를 제공해 사용자가 스스로 검증할 수 있도록 돕는 웹앱을 목표로 합니다.  
또한 비교 결과를 하나의 **객관적 지표(Total Cost Index)**로 제시해 의사결정의 확신을 높입니다.

### 선택 이유
- 기회비용은 널리 알려졌지만, 실제 생활에서는 **숫자로 적용하기 어렵고 기준이 쉽게 흔들리는** 영역입니다.
- 객관적 지표(총비용)를 제공하면 “싸 보이는 선택”이 실제로는 비쌀 수 있음을 빠르게 확인할 수 있어, **의사결정 피로를 줄이고** 기준을 일관되게 만들 수 있습니다.
- 계산 모델이 단순·투명하여 결과를 근거와 함께 제시할 수 있고, 테스트로 검증하기도 쉬워 웹앱 프로토타입에 적합합니다.

### 기능 범위

#### 핵심 기능(MVP)
1) 시간가치(시급) 입력 → 분당 가치 자동 계산  
2) 선택지 A/B 입력: 소요시간(분), 직접비용(원)  
3) 결과 출력: A 총비용, B 총비용, 유리한 선택, 차액(원)  
4) 근거 제공: 직접비용/시간비용 분해 + 계산식(토글)  
5) Total Cost Index(총비용 지표)로 비교 결과를 직관적으로 표시  
6) 프리셋 예시 제공(온보딩)

#### 부가 기능(확장)
- 다안 비교(3~5개 선택지)
- 결정 히스토리 저장(localStorage)
- 피로/스트레스 비용 옵션(가중치 반영)
- 배포(Vercel 등)
- 계산 로직 단위 테스트

### 계산 로직(정의)

#### 핵심 계산 모델
사용자의 시급을 $W$, 소요 시간을 $T$(분), 직접 비용을 $C$라고 할 때 총 비용($TC$)은 다음과 같이 정의합니다.

$$TC = C + \left( \frac{W}{60} \times T \right)$$

- $W$ : 사용자가 입력한 시간당 가치 (원/시간)  
- $T$ : 해당 선택지에 소요되는 총 시간 (분)  
- $C$ : 해당 선택지에 필요한 결제 금액 (원)

> Total Cost Index는 기본적으로 위의 $TC$ 값을 의미하며, A/B의 $TC$를 비교해 더 낮은 선택을 추천합니다.

#### 소수점 처리 기준(원 단위)
- $\frac{W}{60} \times T$ 계산 과정에서 소수점이 발생할 수 있으므로, 최종 총비용은 **원 단위 절삭(floor)** 처리합니다.  
  (예: 5,500.9원 → 5,500원)  
- 절삭 기준은 문서/화면에 명시하여 결과 신뢰성을 유지합니다.

### 대상 사용자
- 직장인/프리랜서/학생 등, “시간이 중요한 자원”인 사용자
- 절약/합리적 소비를 원하지만 시간까지 반영한 기준이 없어 자주 고민하는 사용자
- 선택지가 많아질수록 판단이 어려워 **의사결정 피로**를 느끼는 사용자

### 사용자 시나리오

#### 시나리오 1) 더 싼 마트 vs 가까운 편의점
- 시급: 15,000원 → 분당 250원
- A(편의점): 3,000원 / 10분
- B(마트): 2,300원 / 40분
- A 총비용 = 3,000 + 10×250 = 5,500원
- B 총비용 = 2,300 + 40×250 = 12,300원  
→ 결론: **A가 유리(시간비용이 판단을 바꿈)**

#### 시나리오 2) 직접 요리 vs 배달
- 시급: 20,000원 → 분당 333원
- A(요리): 6,000원 / 60분
- B(배달): 14,000원 / 10분  
→ 직감과 결론이 달라질 수 있어, **근거(시간비용/수식) 표시가 중요**

#### 시나리오 3) 무료배송 기다리기 vs 유료배송 빠르게 받기
- A(무료배송): 비용 0원 / 추가 대기 시간 발생
- B(유료배송): 비용 발생 / 대기 시간 절감  
→ 사용자가 “추가로 낭비되는 시간(분)”을 입력하여 비교

### 엣지 케이스 및 처리
1) 시급 0/음수/문자 입력 → 숫자만 허용, 1 이상 제한, 오류 메시지  
2) 시간 0분 → 허용(시간비용 0으로 계산)  
3) 비용 0원 → 허용(무료 옵션)  
4) A/B 총비용 동일 → “차이 없음(동일)” 표시  
5) 비현실적으로 큰 값 입력 → 경고(입력 실수 가능성 안내), 계산은 유지  

#### 추가 엣지 케이스(단위/표현/인사이트)
6) 입력 단위 혼동(시간: 분 vs 시/분)
- UI에서 시간 입력을 기본적으로 “분”으로 받되, **1시간 20분 → 80분**처럼 자동 변환 입력을 지원(선택)하거나,
- 최소한 placeholder 예시(예: `80 (분)`)로 실수 가능성을 낮춥니다.

7) 시급이 비현실적으로 높은 경우(예: 1억 원/시간)
- 계산은 정상 수행하되, 결과 영역에 인사이트 메시지 출력:
  - 예: “시간의 가치가 매우 큽니다. 빠른 선택이 압도적으로 유리할 가능성이 높습니다.”
  - (과장된 단정 대신 ‘가능성’ 표현으로 신뢰성 유지)

8) 소수점 처리 명시
- 분당 가치/시간비용에서 소수점이 생길 수 있으므로, **원 단위 절삭(floor)** 기준을 고정하고 화면/문서에 명시합니다.

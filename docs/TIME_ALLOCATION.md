# 시간 배분 및 우선순위 조정 (Time Allocation & Priority Adjustments)

## 목적
이 문서는 **실제 시간 배분과 우선순위 조정 내역**을 구체적으로 기록하여, 제한된 시간 내에서의 의사결정 과정을 투명하게 보여줍니다.

## 3. 주요 우선순위 조정 (3회)

### 조정 #1: UI 시간 감소 (Day 1 오후)

**시점**: v0.3 (API 구현 완료 후)

**원래 계획**:
```
API 구현 (2h) → UI 구현 (4h) → 검증 (2h)
```

**변경 후**:
```
API 구현 (2h) → 검증 (1h) → UI 구현 (3h)
```

**조정 이유**:
- API를 먼저 완성하고 검증을 돌려보니, 엣지 케이스가 더 필요함을 발견
- UI를 일찍 만들면 API 스펙 변경 시 UI도 두 번 수정해야 함
- **"API 스펙을 먼저 고정하면 UI 작업이 일회성"**이라는 판단

**효과**:
- UI 재작업 3회 → 1회로 감소 (추정)
- 개발 흐름이 더 예측 가능해짐

**시간 절약**: 약 1시간

---

### 조정 #2: 사용자 피드백 반영 우선 (Day 2 오전)

**시점**: v1.0 (MVP 완성 후)

**원래 계획**:
```
MVP 완성 → 테스트 커버리지 확장 → 배포 → 문서화
```

**변경 후**:
```
MVP 완성 → 사용자 테스트 → 피드백 즉시 반영 → 테스트 → 문서
```

**조정 이유**:
- 실제 사용자 테스트에서 4건의 구체적 피드백 발견
  1. 계산식이 이해하기 어렵다
  2. 다안 프리셋이 작동하지 않는다
  3. 히스토리 정책이 혼란스럽다
  4. 단위 변환이 어렵다
- "가상으로 예상한 문제"보다 "실제 사용자가 보고한 문제"가 훨씬 가치 있음
- **"사용자 관점의 문제 > 개발자 관점의 완성도"** 판단

**효과**:
- UX가 크게 개선됨
- 사용자 시나리오가 추상적 설명 → 검증된 경험으로 전환

**투자 시간**: +2시간 (계획에 없던 항목)

---

### 조정 #3: 배포 실행 → 배포 문서화 (Day 2 오후)

**시점**: v1.3 (사용자 피드백 반영 후)

**원래 계획**:
```
Heroku/Railway에 실제 배포 (예상 1~2시간)
```

**변경 후**:
```
Docker, Heroku, Railway, AWS, GCP 배포 가이드 작성 (30분)
```

**조정 이유**:
- 실제 배포 시도했으나 환경 설정, 계정 문제 등으로 예상보다 오래 걸림
- 배포 URL vs 로컬 실행 가능한 고품질 앱 → **후자가 평가에서 더 중요**하다고 판단
- 배포 문서만 있어도 "언제든 배포 가능한 상태"를 증명 가능

**Trade-off 분석**:
```
[배포 실행에 2시간 투자]
+ 접근성 향상 (URL로 즉시 테스트)
+ 배포 경험
- 기능/테스트 시간 감소
- 배포 실패 시 시간 낭비

[문서화에 30분 투자 + 나머지 1.5h를 품질에]
+ 테스트 강화 (17 → 18개)
+ 문서 정리 (ARCHITECTURE 추가)
+ 피드백 완전 반영
- 배포 URL 없음
```

**최종 선택**: 후자
**근거**: 과제 평가 기준상 "기능 정확성(10점)"이 배포(선택적)보다 중요

**시간 재분배**:
- 배포: 2h → 0.5h (절약 1.5h)
- 테스트: 1h → 1.5h (+0.5h)
- 문서: 1h → 2h (+1h)

## 4. 예상 외 시간 소요 항목

### 4-1. 다안 비교 프리셋 버그 (예상 외 +2h)

**원인**: 초기 설계에서 데이터 구조 확장성 고려 부족

**시도 과정**:
```
[1차 시도] 0.5h
- data-index로 DOM 선택 개선
- 결과: A/B만 채워짐

[2차 시도] 1h
- 반복문 추가
- 결과: C/D/E 여전히 비어있음
- 원인 발견: 데이터 구조 문제

[3차 시도] 0.5h
- options 배열로 확장
- 결과: ✅ 완전 해결
```

**교훈**: 
- 확장 가능한 데이터 구조를 처음부터 설계하면 시간 절약
- 3번 시도한 시간(2h)을 초기 설계에 10분만 더 투자했어도 방지 가능

---

### 4-2. Git 커밋 메시지 전략 변경 (예상 외 +1h)

**원인**: PowerShell에서 heredoc 구문 오류

**시도 과정**:
```
[1차] heredoc 구문 시도 → 파싱 에러
[2차] 단일 라인 메시지로 변경 → 성공
[문서화] CHANGELOG/RETROSPECTIVE 정리
```

**결과**:
- 커밋 메시지: 간략 요약만
- 상세 내용: 문서로 관리
- 오히려 더 깔끔한 구조

**교훈**: "실패"가 더 나은 방향으로 이끌기도 함

---

### 4-3. 시급 상한선 검증 추가 (예상 외 +1h)

**발견 시점**: 사용자 보고
**원인**: 초기 엣지 케이스 분석에서 누락

**해결 과정**:
```
사용자 보고 (15분)
   ↓
원인 분석 (10분) - @Max 부재
   ↓
백엔드 수정 (5분) - @Max 추가
   ↓
프론트 수정 (10분) - 상한선 체크
   ↓
테스트 추가 (10분) - testExceedsMaxWage
   ↓
문서 업데이트 (10분) - TROUBLESHOOTING, CHANGELOG
```

**총 소요**: 1시간
**교훈**: 초기 엣지 케이스 체크리스트가 완벽하지 않았음

## 5. 시간 배분 회고

### 계획 대비 실제

| 작업 | 계획 | 실제 | 차이 | 분석 |
|------|------|------|------|------|
| 로직 | 2h | 3h | +1h | 테스트 포함 시 정확 |
| UI | 4h | 3h | -1h | API 우선 전략 효과 |
| UX 개선 | 0h | 2h | +2h | 사용자 피드백 (계획 외) |
| 검증 | 2h | 1.5h | -0.5h | @Valid로 효율화 |
| 테스트 | 1h | 1.5h | +0.5h | 커버리지 강화 |
| 배포 | 1h | 0.5h | -0.5h | 문서화로 대체 |
| 문서 | 1h | 2h | +1h | 8개 문서 작성 |
| 버그 수정 | 0h | 2h | +2h | 프리셋 3차 시도 |

**총계**: 12h → 15h (+3h)

### 초과 시간 분석

**+3시간의 사용처**:
- 사용자 피드백 반영: 2h (가치 높음)
- 예상 외 버그 수정: 2h (필수)
- 문서 강화: 1h (맥락 관리)
- 테스트 확장: 0.5h (품질 보증)

**정당성**:
- 실제 사용자 문제 해결 > 계획 준수
- 품질 > 일정
- 15시간도 합리적 범위

## 6. 구체적 조정 사례

### 사례 1: "프리셋 vs 테스트" 우선순위 (Day 1 저녁)

**상황**: MVP 완성, 남은 시간 4시간

**선택지**:
- A: 프리셋 6개 추가 (2h) + 테스트 (2h)
- B: 테스트 확장 (4h)
- C: 다안 비교 구현 (4h)

**선택**: A
**이유**: 
- 프리셋은 온보딩 경험에 직접 영향
- 테스트는 필수지만, 11개로도 기본 커버
- 다안 비교는 부가 기능

**결과**: 
- 프리셋으로 사용자 경험 ↑
- 테스트도 기본 수준 확보

---

### 사례 2: "버그 수정 vs 새 기능" (Day 2 오전)

**상황**: 다안 프리셋 버그 발견, 남은 시간 3시간

**선택지**:
- A: 버그 완전 해결 (2h) + 테스트 강화 (1h)
- B: 버그 부분 해결 (1h) + 배포 시도 (2h)
- C: 버그 무시하고 배포/문서 진행

**선택**: A
**이유**:
- "작동하지 않는 기능"은 없느니만 못함
- 배포는 선택적, 기능 정확성은 필수
- 프리셋은 UX의 핵심 요소

**결과**:
- 버그 완전 해결 (3차 시도)
- 테스트 17→18개 확장
- 배포는 문서화로 대체

**비용**: 배포 URL 포기
**가치**: 모든 기능이 완전히 동작

---

### 사례 3: "문서 vs 배포" (Day 2 오후)

**상황**: 마감 6시간 전, 기능은 완성

**선택지**:
- A: Heroku 배포 시도 (2h) + 간단한 README (1h)
- B: 배포 준비 문서 (0.5h) + 상세 문서화 (3h)

**선택**: B
**이유**:
- 배포 실패 시 2시간 낭비 위험
- 과제 평가 기준: "개선 과정 기록(8점)" > "배포 완료(선택)"
- 문서로 "배포 가능 상태"를 증명 가능

**결과**:
- CHANGELOG (30개 버전)
- RETROSPECTIVE (회고)
- TROUBLESHOOTING (17개 항목)
- AI_LOG (AI 활용 기록)
- ARCHITECTURE (구조 설명)
- DEPLOYMENT (5개 플랫폼 가이드)
- API (명세)
- DESIGN_EVOLUTION (신규)

**비용**: 배포 URL 없음
**가치**: 체계적인 문서화로 사고 과정 명확히 전달

## 8. 시간 투자 대비 효과

### 높은 ROI (Return on Investment)

| 투자 | 시간 | 효과 | ROI |
|------|------|------|-----|
| API 우선 전략 | +1h | UI 재작업 3→1회 (절약 2h) | **2배** |
| 사용자 피드백 | +2h | 실제 문제 4건 해결 | **매우 높음** |
| 문서 구조화 | +1h | 맥락 관리, 회고 용이 | **높음** |
| 테스트 확장 | +0.5h | 버그 조기 발견, 안정성 | **높음** |

### 낮은 ROI (하지만 필요)

| 투자 | 시간 | 효과 | 이유 |
|------|------|------|------|
| 프리셋 6개 작성 | 1h | 온보딩 개선 | UX 필수 |
| 배포 문서 5개 | 0.5h | 준비 상태 증명 | 필요 시 즉시 배포 |

### 낮은 ROI (피함)

| 회피 | 절약 | 트레이드오프 |
|------|------|-------------|
| 실제 배포 | 1.5h | 접근성 vs 품질 → 품질 선택 |
| React 전환 | 5h | SPA vs 빠른 MVP → 후자 선택 |
| Redis 도입 | 2h | 분산 캐시 vs 단순성 → 후자 선택 |

## 9. 최종 시간 효율성 분석

### 계획 대비

```
[계획]
```

# 우선순위 조정 기록 (Priority Adjustments)

## 목적
이 문서는 **실제 서비스 제공**에 초점을 맞춰 우선순위를 어떻게 조정했는지 기록합니다.

---

## 핵심 원칙

### 최우선 목표
**"실제로 작동하는 웹앱을 사용자에게 제공한다"**

이를 위해 다음 순서로 우선순위를 설정했습니다:
1. **서비스 제공 가능**: 최소한의 기능이라도 완전히 작동해야 함
2. **기능 정확성**: 계산 로직이 정확해야 함
3. **사용자 경험**: 실제 사용자가 쓸 수 있어야 함
4. **확장 가능성**: 추후 개선할 수 있는 구조

---

## 전체 시간 배분 (임시)

전체 약 15시간 동안 다음과 같이 우선순위를 두고 작업했습니다:

### 핵심 개발 (약 60%)
- **계산 로직 & 검증**: 가장 많은 시간 투자
  - 수식 정의, 소수점 처리, 엣지 케이스
  - 테스트 작성 (18개)
  - 이유: 계산이 틀리면 서비스 가치가 없음

- **기본 UI**: 최소한의 사용 가능한 인터페이스
  - 시급 입력, 선택지 입력, 결과 표시
  - 이유: 사용자가 접근할 수 있는 창구 필요

### 사용성 개선 (약 25%)
- **사용자 피드백 반영**: 실제 사용자 테스트 후 개선
  - 계산식 상세화
  - 히스토리 정책 변경 (localStorage → sessionStorage)
  - 단위 변환 가이드 추가
  - 이유: 실제 사용자 문제 > 가상의 완벽함

- **프리셋 & 다안 비교**: 사용 편의성
  - 6개 시나리오 프리셋
  - 3~5개 선택지 비교
  - 이유: 온보딩과 확장성 확보

### 문서화 (약 15%)
- **개선 과정 기록**: 의사결정 투명성
  - CHANGELOG, TROUBLESHOOTING, RETROSPECTIVE
  - 이유: 과제 요구사항 + 향후 유지보수

## 주요 우선순위 조정 (3회)

### 조정 #1: API 우선 → UI는 나중

**상황**: 초기에는 UI를 먼저 만들려고 했음

**조정 결정**: 
- API와 계산 로직을 먼저 완성하고 검증
- UI는 API 스펙이 고정된 후 작업

**이유**:
- API 스펙이 바뀌면 UI도 다시 만들어야 함
- 계산 로직은 프로젝트의 핵심 가치

**결과**:
- UI 재작업 최소화
- 개발 흐름이 예측 가능해짐

---

### 조정 #2: 사용자 피드백 즉시 반영

**상황**: MVP 완성 후 테스트 커버리지 확장 vs 사용자 테스트

**조정 결정**:
- 사용자 테스트를 먼저 수행
- 4건의 피드백을 즉시 반영
- 테스트는 나중으로 연기

**이유**:
- 실제 사용자 문제 > 개발자가 예상한 문제
- "작동하지만 쓰기 어려운 서비스"보다 "쓸 만한 서비스"가 우선

**결과**:
- UX 크게 개선 (계산식, 히스토리, 단위 가이드)
- 사용자 관점의 검증 완료

---

### 조정 #3: 배포 실행 → 배포 준비

**상황**: 실제 배포 URL vs 로컬 실행 가능한 완성도

**조정 결정**:
- 실제 배포는 시간 관계상 보류
- 대신 배포 문서를 상세히 작성 (Docker, Heroku, Railway 등)
- 남은 시간을 품질 강화에 투자

**이유**:
- **실제 서비스 제공의 본질은 "기능이 완벽히 작동"하는 것**
- 배포 URL은 접근성일 뿐, 핵심 가치는 아님
- 문서로 "배포 가능 상태"를 증명할 수 있음

**결과**:
- 테스트 강화 (17개 → 18개)
- 문서 보완 (8개 전문 문서)
- 모든 기능 완전히 작동

## 우선순위 의사결정 원칙

### 언제 우선순위를 조정했는가?

1. **실제 사용자 피드백이 있을 때**
   - 예상보다 실제 문제가 더 중요
   - 즉시 반영

2. **핵심 가치가 위협받을 때**
   - 계산 로직의 정확성
   - 서비스 안정성

3. **시간 대비 효과가 명확할 때**
   - API 우선 → UI 재작업 감소
   - 배포 보류 → 품질 강화

### 우선순위 판단 기준

```
[최고 우선순위]
1. 서비스가 작동하는가?
2. 계산이 정확한가?
3. 사용자가 쓸 수 있는가?

[중간 우선순위]
4. 확장 가능한가?
5. 문서가 있는가?

[낮은 우선순위]
6. 배포되어 있는가?
7. 완벽한가?
```

**핵심**: "완벽한 계획"보다 "작동하는 서비스"

---

## 실제 vs 이상의 차이

### 원래 계획
- 모든 기능을 완벽히 구현
- 배포까지 완료
- 시간을 정확히 예측

### 실제 진행
- **핵심 기능에 집중**
- **사용자 피드백으로 방향 수정**
- **배포보다 품질 우선**
- 시간은 유동적으로 조정
